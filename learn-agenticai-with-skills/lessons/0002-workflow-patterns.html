Workflow Patterns

The five canonical ways to compose multiple LLM calls into a system

In Lesson 1 you drew the line between workflows and agents. Now we populate the workflow side of that line. Workflows are where most production systems live — and the five patterns below cover the vast majority of what you'll encounter and design.

Core premise
All five patterns below are workflows: the developer defines the control flow. The LLM does text transformation inside nodes — it never chooses the next step.

Source: Anthropic, "Building Effective Agents" (2024)

1 · Prompt Chaining

When: the task decomposes cleanly into a fixed sequence of dependent steps

Each LLM call's output becomes the next call's input. You add quality gates between steps — a programmatic check that aborts the chain if an intermediate result is wrong before wasting compute on downstream steps. [Anthropic, 2024]

Input ──► LLM₁ ──►[gate]──► LLM₂ ──►[gate]──► LLM₃ ──► Output ✗ abort ✗ abort
Each step transforms the output of the previous one. Gates catch failures early.
+ Failures are localised — you know exactly which step broke
+ Simpler to test and reason about than a monolithic prompt
− Latency is additive — steps run sequentially
− Errors propagate: a bad step 1 poisons every step downstream

2 · Routing

When: inputs fall into distinct categories requiring different specialised handling

A classifier (LLM or traditional ML) examines the input and directs it to the appropriate specialised chain or prompt. Each route is optimised for its category rather than trying to handle everything with a single generalised prompt. [Anthropic, 2024]

┌──► Route A (e.g. billing query) │ Input ──► Classifier ──► Route B (e.g. tech support) │ └──► Route C (e.g. cancellation)
One classifier, many specialised downstream paths.
+ Specialised prompts outperform generalised ones per category
+ Easy to add new routes without touching existing ones
− Classifier errors send inputs down the wrong path
− Ambiguous inputs may span multiple categories cleanly

3 · Parallelisation

When: tasks are independent, or you need higher confidence via multiple perspectives

Two sub-patterns share the same structure but serve different goals:

Sectioning — split a task into independent chunks and run them simultaneously. A long document gets processed in parallel sections; a complex task gets concurrent specialised workers. Reduces wall-clock latency when parts don't depend on each other.

Voting — run the same task multiple times independently, then aggregate. Used when a single LLM response is unreliable: majority vote increases confidence. [Anthropic, 2024]

Sectioning Voting Input Input │ │ ├──► LLM (chunk A) ──┐ ├──► LLM run 1 ──┐ ├──► LLM (chunk B) ──┤ aggregate ├──► LLM run 2 ──┤ majority └──► LLM (chunk C) ──┘ └──► LLM run 3 ──┘ │ │ Output Output
Left: reduces latency by splitting. Right: increases confidence by repeating.
+ Sectioning cuts latency proportionally to parallelism
+ Voting surfaces high-confidence answers and catches hallucinations
− Requires that tasks are genuinely independent (sectioning)
− Voting multiplies cost by N runs

4 · Orchestrator-Workers

When: the full task can't be decomposed in advance — the plan must be constructed dynamically

An orchestrator LLM analyses the task and dynamically dispatches subtasks to worker LLMs. Workers complete their subtasks and report back; the orchestrator synthesises results and may spawn more workers. This looks like parallelisation, but with a critical difference: the orchestrator decides at runtime what work to create. [Anthropic, 2024]

┌──────────────────────────────┐ │ Orchestrator │ │ (decides what work to do) │ └──────────┬───────────────────┘ │ dispatches ┌──────────┼──────────┐ ▼ ▼ ▼ Worker A Worker B Worker C │ │ │ └──────────┴──────────┘ │ results ▼ Orchestrator synthesises
The orchestrator is a developer-defined router — control flow is still pre-coded, but the routing decisions use LLM reasoning.
+ Handles open-ended tasks where steps aren't knowable upfront
+ Worker specialisation is clean — each worker does one thing
− Cost and latency are harder to bound than fixed chains
− Orchestrator reasoning errors cascade to all workers

5 · Evaluator-Optimizer

When: output quality can be assessed by a separate LLM, and iteration improves quality

One LLM generates a response; a second LLM evaluates it against a rubric and produces feedback. If the feedback identifies deficiencies, the output is revised and re-evaluated. This loops until quality criteria are met or a maximum iteration count is reached. [Anthropic, 2024]

┌─────── feedback ◄────────────────┐ │ │ ▼ │ Input ──► Generator ──► output ──► Evaluator ─┤ │ └── "pass" ──► Output
The loop runs until the evaluator is satisfied. The exit condition must be explicit to prevent infinite loops.
+ Iterative refinement consistently outperforms single-shot generation
+ Evaluation criteria are explicit and testable
− Cost and latency scale with iteration count
− Evaluator and generator can reach a local optimum that still fails

Choosing a Pattern

In practice, these patterns compose. A routing step at the front can split inputs between a prompt chain and an evaluator-optimizer loop. An orchestrator-workers step can use sectioning parallelisation internally. The question is always: what is the most constrained thing I need the control flow to do?

Task requires fixed sequence of dependent steps? → Prompt Chaining Task inputs fall into known, distinct categories? → Routing Task splits into independent, parallelisable parts? → Parallelisation (Sectioning) Need higher confidence from a single task? → Parallelisation (Voting) Task structure is unknown until you start? → Orchestrator-Workers Output quality can be assessed and iterated on? → Evaluator-Optimizer
These are starting points, not rigid rules. Most systems combine more than one.
Boundary reminder
None of the five patterns above crosses into agent territory. An orchestrator-workers system still has developer-defined control flow — the orchestrator is a node in a graph you designed, not an LLM that has autonomously decided to spawn workers. The agent boundary is crossed when the LLM dynamically decides its own next action without the developer pre-coding that decision point.

Retrieval check — pattern recognition

A code documentation tool first extracts all function signatures from a file (step 1), then generates a docstring for each signature (step 2), then writes a module-level summary using the docstrings as input (step 3). Each step requires the previous one. Which pattern is this?

  • Routing — classifier directs to specialised handling
  • Prompt chaining — sequential dependent steps
  • Parallelisation — independent tasks run concurrently
  • Evaluator-optimizer — generate then critique in a loop

Retrieval check — pattern recognition

A research assistant receives a broad question, splits it into four independent sub-questions, dispatches all four to separate LLM calls at the same time, then synthesises the answers into a final response. Which pattern is this?

  • Prompt chaining — sequential dependent steps
  • Orchestrator-workers — dynamic task allocation
  • Parallelisation — independent tasks run concurrently
  • Evaluator-optimizer — generate then critique in a loop

Retrieval check — pattern recognition

A writing assistant generates a draft essay, then a second LLM scores it against a clarity rubric and returns critique. If the score is below threshold, the draft is revised and re-scored. This repeats until the rubric is satisfied. Which pattern is this?

  • Prompt chaining — sequential dependent steps
  • Routing — classifier directs to specialised handling
  • Orchestrator-workers — dynamic task allocation
  • Evaluator-optimizer — generate then critique in a loop
