Memory Systems

Where agents store and retrieve what they know — and why the tradeoffs are architectural

In Lesson 3 you saw that the ReAct loop accumulates its entire trace in the context window. That's fine for short runs. But real agents need to remember things across sessions, across users, and at scales that dwarf any context window. Memory system design is how you handle that.

The most rigorous framework for reasoning about agent memory comes from the CoALA survey (Cognitive Architectures for Language Agents). It identifies four types of memory, each with distinct properties and tradeoffs. [Sumers et al., 2023]

The Central Tension

The context window is the only space where reasoning happens. The model cannot reason about anything outside it. This creates an irreducible tension:

More in-context → richer reasoning, but token cost grows and context fills Retrieve externally → scalable storage, but retrieval errors corrupt the reasoning In weights → zero retrieval cost at inference, but expensive to update Episodic logs → agents that learn from experience, but require retrieval infrastructure
Every memory design decision navigates this tension. There is no dominant strategy.
The core insight
The context window is the agent's entire working memory. Every other memory type is external to it — and must be retrieved into it before the model can reason about it. Retrieval is never free: it consumes tokens, takes time, and can be wrong.

The Four Memory Types

In-Context (Working Memory)

The active context window. Everything the model can currently attend to.

  • System prompt, conversation history, tool results, retrieved docs
  • The entire ReAct trace accumulates here
  • Cleared at session end — nothing persists automatically
  • Hard ceiling: context window size
zero retrieval error instant access finite ephemeral

External (Persistent) Memory

Stored outside the model; retrieved into context on demand.

  • Fuzzy retrieval: vector similarity search (semantic matching)
  • Exact retrieval: SQL, keyword, hash lookups
  • Persists across sessions; scales to millions of documents
  • Retrieval can return wrong or incomplete chunks
persistent scalable retrieval error risk latency cost

Episodic Memory

Records of past agent runs — what the agent did, with what results.

  • Enables few-shot prompting from real, verified task runs
  • Supports debugging: replay the exact trace that produced a failure
  • Enables improvement over time (agent learns from its own history)
  • Implementation: store trace embeddings; retrieve by similarity to current task
experiential persistent retrieval required storage overhead

Semantic Memory

Factual world knowledge. Exists in two locations:

  • In-weights: baked in during training/fine-tuning. Zero retrieval cost; stale after cutoff; expensive to update.
  • Externalized (RAG): stored in a knowledge base, retrieved at runtime. Updateable; subject to retrieval errors and latency.
in-weights: fast in-weights: stale RAG: updateable RAG: retrieval errors

The Fine-Tuning vs RAG Decision

For semantic memory, the most common design question is whether to bake knowledge into the model (fine-tuning) or retrieve it at inference time (RAG). This is not a binary choice — many production systems do both — but the tradeoffs are distinct:

Dimension Fine-tuning (in-weights) RAG (externalized)
Inference cost Zero — knowledge is in the model Retrieval step + embedding cost
Update cost High — requires retraining or fine-tuning Low — update the knowledge base
Update latency Days to weeks (training pipeline) Near-real-time
Accuracy risk Catastrophic forgetting; knowledge blending Retrieval error; wrong chunks retrieved
Best for Stable, high-frequency knowledge (writing style, domain syntax) Dynamic, large, or user-specific knowledge
Rule of thumb
Fine-tune for how to reason (style, format, domain syntax). Use RAG for what to know (facts, documents, user data). The distinction maps roughly onto procedural vs semantic memory.

Source: Sumers et al., "CoALA" (2023)

Memory in the ReAct Loop

Every step of the ReAct loop interacts with these memory types. A realistic single step looks like this:

Before the loop starts: External memory ──► retrieval ──► loaded into context (in-context) Episodic memory ──► relevant past run ──► loaded as few-shot example (in-context) During the loop: Thought / Action / Observation ──────────────────► appended to in-context Tool call (read_file, query_db) ──► result ──────► Observation added to in-context After the loop ends: Entire trace ──► stored to episodic memory (for future runs) New facts extracted ──► stored to external memory (knowledge base update)
In-context memory is the integration point. Everything retrieved from other memory types flows into the context window before the model can reason about it.

What Goes Where: Design Heuristics

When designing a new agent, these questions determine the memory architecture:

  • Does it fit in context? If yes, and you need the model to reason over all of it simultaneously, load it in-context. Retrieval introduces error; in-context doesn't.
  • Is it user-specific or entity-specific? Use external memory — it can't be fine-tuned per user.
  • Does it change frequently? External memory (RAG) over fine-tuning. Stale weights are a silent correctness failure.
  • Do you need the agent to improve from experience? Episodic memory — store successful traces; retrieve them as few-shot examples for similar future tasks.
  • Is it stable, high-frequency background knowledge? In-weights via fine-tuning — zero inference-time cost, never retrieved incorrectly.

Retrieval check — memory type identification

A coding agent is given a 40-page internal API reference at the start of every session. It is included verbatim in the system prompt and available throughout the run. Which memory type is this?

  • In-context — loaded directly into the active prompt
  • External — fetched from a vector store on demand
  • Episodic — replayed from a record of past sessions
  • Semantic — encoded into the model at training time

Retrieval check — tradeoff reasoning

A support agent must look up a customer's 8 recent orders (500 tokens total) before responding. The data fits the context window. What is the strongest argument for loading it directly into context rather than retrieving it on demand?

  • It eliminates the risk of retrieving incorrect chunks
  • It reduces the round-trip latency for each response
  • It avoids the cost of maintaining a vector database
  • It ensures the model can see every order record

Retrieval check — connecting memory to the ReAct loop

An evaluator-optimizer workflow runs 8 iterations. Each evaluator critique is appended to the context before the next generator pass. In iteration 8, the generator contradicts a constraint the evaluator established in iteration 2. What is the most likely cause?

  • Prior evaluations were truncated when the context window filled
  • The evaluator received a different system prompt each round
  • Evaluation logs were not persisted to external storage systems
  • The generator model was not updated between iteration rounds
