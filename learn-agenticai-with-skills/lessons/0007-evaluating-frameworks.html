Lesson 07 · Agentic AI

Evaluating Agent Frameworks

A six-lens method for reading any new framework, paper, or codebase against your mental model

The agentic AI space produces new frameworks every few months — AutoGen, LangGraph, CrewAI, Semantic Kernel, and more. Each arrives with marketing claims about autonomy, multi-agent capability, and production-readiness. Without a structured lens, you have no basis for evaluating those claims beyond surface impressions.

The prior six lessons are that lens. This lesson turns them into a repeatable evaluation method.

The Six-Lens Framework

Each lens maps to one lesson. Apply them in order — each one narrows the question before the next one goes deeper.

Lens Core question Red flag
L1 · Workflow vs Agent Does the framework's "agent" give control to the LLM, or does the developer still define the flow? Is the distinction clearly drawn? Calls everything "agents" regardless of who controls flow. Blurs the distinction in marketing.
L2 · Patterns Which of the five workflow patterns does it natively support? Which require awkward workarounds? Can you implement evaluator-optimizer without hacking the internals? Only one topology. No first-class evaluator loop. Forces all tasks into one pattern shape.
L3 · ReAct loop How does the harness inject observations? Is the loop structure inspectable in traces? Are there built-in step limits and loop detection? No step limit. No loop detection. Opaque traces that can't be reconstructed after the fact.
L4 · Memory What memory systems are available? In-context only, or external stores? What happens when context fills — is there rolling summarisation, or a hard crash? No mechanism for context exhaustion. External memory bolted on as an afterthought with no clear retrieval strategy.
L5 · Failure modes What harness-level mitigations are built in? Is there prompt injection hardening? Are tools scoped per agent or shared globally? No tool scoping. No prompt injection guidance in the docs. Assumes model self-protection.
L6 · Multi-agent trust How are worker scopes defined — at system prompt construction, or by runtime assertion? What topologies are natively supported vs. requiring custom code? Peer-network as the default topology with no structured alternatives. Runtime trust claims accepted without question.
Using the lens: you don't need all six answers before forming an opinion. If L1 reveals that the framework mislabels workflows as agents, that shapes how much you trust the L2–L6 claims. Start with L1 and go as deep as the decision warrants.

Worked Example: AutoGen

AutoGen (Wu et al., 2023) is a Microsoft Research multi-agent conversation framework. It was cited in Lesson 6's source box as a concrete peer-network implementation. Here is the six-lens read.

AutoGen

Wu et al., 2023 · Microsoft Research · arxiv.org/abs/2308.08155
L1 · Workflow vs Agent
Mixed AutoGen distinguishes AssistantAgent (LLM-driven) from UserProxyAgent (code-executor, human-in-the-loop). The distinction maps reasonably to agent vs. workflow node. However, UserProxyAgent is often used in roles where a human never actually participates — the name implies human oversight that may not exist at runtime.
L2 · Patterns
Weak The primary topology is peer-to-peer conversation. Sequential pipelines and fan-out/fan-in are supported but require custom orchestration code on top of the conversation primitives. The evaluator-optimizer pattern has no native representation — you would need to implement the loop and rubric yourself.
L3 · ReAct loop
Strong AutoGen does include max_consecutive_auto_reply — a built-in step limit per agent. This directly mitigates action loops. Conversation history serves as the trace and is inspectable. Observations (code execution results) are injected into the conversation as UserProxyAgent turns.
L4 · Memory
Weak Memory is the conversation history — in-context only by default. No built-in rolling summarisation for long conversations. External memory (vector stores, etc.) is not integrated into the core framework; it requires custom retrieval logic. Context exhaustion is a real risk in long multi-agent conversations.
L5 · Failure modes
Weak No built-in prompt injection hardening in the core framework. Tools (functions) are shared at the agent level — no fine-grained tool scoping per workflow step. Minimal footprint is a design responsibility that falls entirely on the implementer.
L6 · Multi-agent trust
Weak Peer-network is the default and primary topology. Agent system prompts can be customised (which is the correct way to set scope), but the framework does not enforce or encourage per-agent trust boundaries. A worker agent has no native mechanism to reject instructions from a peer based on scope.

The evaluation yields a clear picture: AutoGen's strength is rapid prototyping of multi-agent conversations. Its weaknesses are in production hardening — memory management, prompt injection, trust boundaries, and topology flexibility. Knowing this, you can adopt it for research experiments while building the mitigations yourself, or choose a different framework for a production deployment where those properties matter.

The key skill
An evaluation does not tell you whether a framework is good or bad. It tells you which properties it trades away for which gains. AutoGen trades production hardening for conversation flexibility. That is a reasonable trade for its intended context (research, rapid prototyping). The lens makes the trade explicit.

Evaluating Papers

The same six lenses apply when reading a research paper. The framing shifts slightly: papers describe systems; frameworks ship systems. For a paper, ask:

  • Which claims match which lens? Map each contribution claim to L1–L6.
  • Which lenses are absent? A paper that never addresses failure modes (L3, L5) has either not tested at scale or left it to future work.
  • What is the evaluation methodology? An evaluator-optimizer paper (L2) that uses a single proxy metric for its rubric is itself vulnerable to specification gaming (L5).

Retrieval check — which lens first

A new framework markets itself as "fully autonomous multi-agent orchestration." Which question should you ask first to evaluate that claim?

  • Whether the LLM or developer controls the flow
  • How many workflow patterns the framework supports
  • What memory systems the framework exposes agents
  • Whether the framework detects prompt injection runtime

Retrieval check — harness gap

A framework's execution harness has no step limit and no repeat-action detection. A coding agent gets stuck on a failing test. What will happen?

  • Agent loops indefinitely whenever it encounters a blocking error
  • Agent loses track of context as the window fills up
  • Agent outputs optimized for a proxy rather than real goal
  • Agent trusts injected instructions arriving through tool results

Retrieval check — trust boundary design

A framework routes all orchestrator messages to workers as user-turn messages, with no system prompt constructed per worker. What principle does this violate?

  • Workers cannot independently validate orchestrator claims at runtime
  • Worker scope must be defined in the system prompt
  • Orchestrators should never pass user messages directly to workers
  • Workers should escalate all ambiguous orchestrator instructions to humans
