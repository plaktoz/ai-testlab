Workflows vs Agents

The foundational distinction that governs every architecture decision

Before committing to any agent architecture, you need to answer one question cleanly: is this a workflow or an agent? The answer determines your reliability guarantees, latency profile, cost model, and debugging story. Most systems that call themselves "agents" are actually workflows — and that's usually the right call.

The Core Distinction

The difference is about who controls the control flow:

┌─────────────────────────────────────────────────────────┐ │ WORKFLOW │ │ │ │ Step A ──► Step B ──► Step C ──► Output │ │ │ │ Control flow: defined by the developer │ │ LLM role: text transformation at each step │ │ Predictability: high (same input → same path) │ └─────────────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────────────┐ │ AGENT │ │ │ │ Input ──► LLM ──► decides next action │ │ ▲ │ │ │ │ ▼ │ │ observe tool / subagent / stop │ │ │ │ │ │ └───────────┘ │ │ │ │ Control flow: determined by the LLM at runtime │ │ LLM role: decision-maker │ │ Predictability: lower (path depends on reasoning) │ └─────────────────────────────────────────────────────────┘
Workflows hard-code control flow. Agents delegate control flow to the LLM.
Key definition
An agent is a system where an LLM dynamically determines its own next action — including whether to use a tool, call a subagent, or stop. A workflow is a system where the developer pre-defines the sequence of LLM calls and logic.

Source: Anthropic, "Building Effective Agents" (2024)

Why This Matters More Than You Think

If you already work with LLMs, you've likely built both without drawing the distinction. The architectural difference has sharp implications:

  • Reliability. In a workflow, you can test every path exhaustively. In an agent, the LLM can take novel paths you never anticipated. This isn't inherently bad — it's why agents can generalize — but it makes verification much harder.
  • Error propagation. In a workflow, you know exactly where a failure occurred. In an agent, errors compound across steps, and earlier reasoning mistakes can corrupt every downstream action. [Anthropic, 2024]
  • Cost and latency. Agents loop; the number of LLM calls is unknown at design time. Workflows have deterministic cost profiles.
  • Debuggability. A workflow failure produces a clear trace. An agent failure often requires reconstructing why the LLM reasoned itself into a bad state.

The Augmented LLM: What Both Are Built From

Both patterns compose from the same building block: an augmented LLM — a model enhanced with retrieval, tools, and memory. The difference is only in how you wire multiple calls together.

Augmented LLM ═══════════════ Context window ┌──────────────────────────────┐ │ System prompt │ │ Conversation history │ │ Retrieved documents (RAG) │ │ Tool results │ └──────────────────────────────┘ │ ▼ Model generates: • text response, OR • tool call(s), OR • both │ ▼ External capabilities ┌──────────────────────────────┐ │ Tools (execute actions) │ │ Retrieval (query memory) │ │ Storage (persist state) │ └──────────────────────────────┘
The augmented LLM — the atom from which all agent and workflow architectures are built.

A key insight: the context window is the LLM's working memory. Everything the model knows about the current task must fit in it, either directly or via retrieval. This constraint shapes almost every architectural decision in agentic systems.

When to Use Each

Anthropic's guidance: "Start with simple prompts and add complexity only when simpler solutions fall short." [Anthropic, 2024]

Use a workflow when:

  • The task is well-defined and the steps are known in advance
  • Reliability and predictability are critical
  • You need deterministic cost/latency
  • You can exhaustively test the paths

Use an agent when:

  • The task requires dynamic decision-making that can't be anticipated
  • The number or type of steps varies significantly across inputs
  • Generalizing to novel situations matters more than strict predictability
  • The cost of human-in-the-loop for each decision is too high
Practical heuristic
If you can flowchart the steps, it's a workflow. If you can't — if the LLM needs to figure out the steps itself — it's an agent.

A Note on the Spectrum

In practice, most production systems are hybrid: a workflow that delegates one or more steps to an agent. For example, a document processing pipeline (workflow) that calls an agent to handle ambiguous edge cases. Knowing the distinction lets you be intentional about where you place the agentic boundary — and therefore where your reliability surface lives.

Retrieval check

A customer support system routes incoming tickets to one of three specialized LLM chains based on a classifier. Each chain follows a fixed sequence of steps to resolve the ticket. What is this?

  • An agent — it uses multiple LLM calls
  • A workflow — control flow is developer-defined
  • An agent — it makes decisions about routing
  • Neither — it requires human-in-the-loop

Retrieval check

What property of the LLM context window most directly constrains agentic system design?

  • Token cost — agents generate more tokens
  • Latency — each tool call adds round-trip time
  • Size — it is the LLM's working memory, so everything must fit in it
  • Immutability — you cannot append to context mid-run
