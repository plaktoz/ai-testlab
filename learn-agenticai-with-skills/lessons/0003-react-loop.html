Lesson 03 · Agentic AI

The ReAct Loop

How agents think, act, and update — the micro-level pattern inside every agent

Lesson 1 defined the agent boundary: an LLM that dynamically determines its own next action. Lesson 2 covered the five patterns on the workflow side. Now we go inside the agent and ask: what does that dynamic control actually look like step by step?

The dominant answer, formalized in 2022, is ReAct.

The Problem ReAct Solves

Before ReAct, two approaches existed — and both were insufficient alone:

  • Chain-of-thought (reasoning only). The model reasons through a problem in natural language. Problem: it can hallucinate facts it doesn't have and can't correct itself when the world contradicts its assumptions. [Wei et al., 2022]
  • Action-only sequences. The model executes a sequence of tool calls. Problem: it can't reason about what it's seeing, so it can't adapt when results are unexpected or when the plan needs to change.

ReAct interleaves both. After each action, the model reasons about the observation before deciding on the next action. This creates a grounded reasoning trace: every thought is anchored to something the model actually observed, not something it assumed. [Yao et al., 2022]

The Three Primitives

A ReAct trace is composed of exactly three repeating components:

Thought — the model's internal reasoning about what to do next Action — a tool call (or a decision to stop and return an answer) Observation — the result returned by the environment after the action
Thought and Action are generated by the model. Observation is returned by the environment.
Key insight
The model writes Thoughts and Actions. It reads Observations. The Observation is not part of the model's generation — it is injected into the context by the system running the agent. This is the concrete mechanism by which the environment updates the model's reasoning.

The Loop Structure

┌─────────────────────────────┐ │ context │ │ system prompt │ │ task │ │ [prior thought/action/obs...] │ └──────────────┬──────────────────┘ │ ▼ ┌──────────────────┐ │ LLM generates │ └──────┬───────────┘ │ ┌───────────────┴───────────────┐ ▼ ▼ Thought (text) Action (tool call) │ │ │ ▼ │ ┌─────────────────────┐ │ │ environment runs │ │ │ tool; returns │ │ │ Observation │ │ └──────────┬──────────┘ │ │ └───────────────┬───────────────┘ │ ▼ append to context; loop (or stop if action = finish)
The loop runs until the model calls a finish action. At each step, the entire prior trace is in the context window.

A Concrete Trace

Here is a realistic ReAct trace for an agent tasked with diagnosing a production incident. The interleaving of reasoning and observation is what allows the agent to change its theory as evidence arrives.

Task: Why is the API returning 500 errors?

Thought: Start with the error logs from the last hour to understand the error type.
Action: read_logs(since="1h", level="error")
Observation: DatabaseConnectionError: max connections exceeded (current: 100, limit: 100)

Thought: Connection pool is exhausted. Check what's holding those connections open.
Action: query_db("SELECT count(*), state FROM pg_stat_activity GROUP BY state")
Observation: active: 87, idle: 13

Thought: 87 active connections is abnormal. Check for long-running queries holding connections.
Action: query_db("SELECT pid, now()-query_start AS dur, query FROM pg_stat_activity WHERE state='active' ORDER BY dur DESC LIMIT 3")
Observation: pid 1234, dur: 47 minutes, query: "SELECT * FROM events WHERE timestamp > '2024-01-01'"

Thought: One runaway query (pid 1234, 47 min) is monopolising connections. I have the root cause.
Action: finish("Root cause: runaway query pid 1234 (47 min) exhausting the connection pool. Fix: terminate it with pg_terminate_backend(1234), then add a query timeout.")
Thought — model's reasoning
Action — tool call
Observation — environment result

Notice what the interleaving buys: after the first observation, the model updated its theory (connection exhaustion → long-running query). Without the Thought step, a pure action sequence couldn't update its plan based on what it saw. The trace is also auditable: every action is preceded by a stated reason.

ReAct and the Agent Boundary

ReAct is the micro-level structure of what it means to be an agent. The loop itself is developer-coded — the system that runs the model and injects observations is your code. But the content of each Thought and the choice of each Action are determined by the LLM at runtime. That's what makes it an agent, not a workflow.

The orchestrator-workers pattern from Lesson 2 typically uses ReAct internally for the orchestrator step. The evaluator-optimizer pattern can also use it in the generator step. ReAct is the foundation; the five workflow patterns are the scaffold you build around it.

Known Failure Modes

Understanding why ReAct succeeds requires knowing where it breaks. Three common failures:

  • Context exhaustion. Every step appends to the context window. Long traces fill it; earlier observations get truncated or summarised, and the model starts contradicting its own prior reasoning. [Sumers et al., 2023]
  • Action loops. When stuck, models sometimes repeat the same action with the same parameters. Without a loop-detection check in the execution harness, the agent runs indefinitely.
  • Reasoning drift. Over many steps, the Thought trace drifts away from the original task. The model optimises for the apparent goal in its recent context rather than the original task specification. Long-horizon tasks are most vulnerable.
Design implication
Each of these failures has a mitigation — but all require the execution harness (your code) to enforce them: context summarisation, loop detection, and step limits. The model cannot protect itself from them. This is a recurring theme in agentic system design.

Retrieval check — trace components

In the trace above, the line "Connection pool is exhausted. Check what's holding those connections open." is an example of which ReAct component?

  • Thought — the model reasons before its next action
  • Action — the model calls a tool in the environment
  • Observation — the environment returns a result
  • Finish — the model returns its final answer

Retrieval check — why interleaving works

What does inserting a Thought step after each Observation give the model that a pure action sequence lacks?

  • A chance to update its plan based on what it observed
  • Faster execution by batching several tools at once
  • Protection against the context window filling up
  • Fewer total tokens consumed across the whole trace

Retrieval check — failure modes

A ReAct agent runs 50 steps on a long research task. Towards the end, its Thought steps start contradicting facts it established in step 5. What is the most likely cause?

  • Early observations were truncated as the context window filled
  • The finish action was not defined in the system prompt
  • Parallel tool calls corrupted the action sequence ordering
  • The model's reasoning tokens exceeded the output limit
