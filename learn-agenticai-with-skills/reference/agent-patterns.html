Agent Patterns — Quick Reference

The foundational taxonomy: workflows, agents, workflow patterns, and the ReAct loop

The Core Distinction → Lesson 1

Workflow Agent
Control flow Developer-defined at design time LLM decides at runtime
LLM role Text transformer at fixed nodes Decision-maker at every step
Predictability High — same input, same path Lower — path depends on reasoning
Debuggability Clear failure localisation Requires trace reconstruction
Cost / latency Deterministic, bounded Variable, unbounded loops
Use when Task steps are known in advance Steps can't be pre-defined
Test: If you can flowchart the steps, it's a workflow. If the LLM must figure out the steps itself, it's an agent.

The Five Workflow Patterns → Lesson 2

1 · Prompt Chaining

When: fixed sequence of dependent steps
  • Each output feeds the next input
  • Add gates to abort on bad intermediate output
  • Trade-off: sequential latency vs. localised failures

2 · Routing

When: distinct input categories need specialised handling
  • Classifier (LLM or ML) directs to the right path
  • Each route is optimised for its category
  • Trade-off: specialisation vs. classifier error risk

3 · Parallelisation

When: tasks are independent or you need higher confidence
  • Sectioning — split task, run chunks simultaneously
  • Voting — run same task N times, aggregate
  • Trade-off: latency/confidence vs. cost × N

4 · Orchestrator-Workers

When: task structure is unknown until you start
  • Orchestrator LLM dynamically dispatches subtasks
  • Worker LLMs execute single specialised subtasks
  • Trade-off: flexibility vs. bounded cost/latency

5 · Evaluator-Optimizer

When: output quality can be assessed by a separate LLM, and iteration improves quality
  • Generator produces output; Evaluator scores against rubric
  • Loop until quality threshold met or max iterations reached
  • Trade-off: highest quality vs. highest cost and latency
  • Critical: the exit condition must be explicit — no implicit "good enough"

The ReAct Loop → Lesson 3

The micro-level structure inside every agent step. Source: Yao et al., 2022

Thought
Model's internal reasoning about the current state and what to do next. Generated by the model. Not an action.
Action
A tool call or the decision to finish. Generated by the model. Executed by the environment.
Observation
The result returned by the environment after the action. Injected into the context by the harness — not generated by the model.
Task → [Thought → Action → Observation] × N → finish(answer)
The loop runs until finish() is called. The full trace accumulates in the context window.

ReAct Failure Modes

FailureCauseMitigation
Context exhaustion Trace fills the context; early observations truncated Rolling summarisation of old trace segments
Action loops Model repeats the same action when stuck Loop detection in the execution harness
Reasoning drift Recent context dominates; model drifts from original task Step limits; task re-injection every N steps
Key asymmetry: the model cannot protect itself from any of these failures. All mitigations live in the execution harness — your code.
Source: Anthropic, "Building Effective Agents" (2024) · Yao et al., ReAct (2022) · Sumers et al., CoALA (2023)
